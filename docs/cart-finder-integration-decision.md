# species-finder "담기" 연동 — 결정 문서

> 목적: 대체수종 장바구니의 마지막 조각(Task 7, species-finder 화면에서 담기)을
> 어떻게 구현할지 결정하기 위한 문서. 구조적 충돌과 3가지 방향을 비교한다.
>
> ⚠️ 현재 상태: 핵심 장바구니 루프(Task 1~6)는 **완성·빌드 통과**.
> simulation 화면에서 담기 → 비교 → 후보 교체 → 확정·저장 → 새로고침 유지 동작.
> 이 문서가 다루는 건 "finder 화면에서도 담을 수 있게 할 것인가, 한다면 어떻게"뿐이다.

---

## 1. 무엇이 문제인가 (구조적 충돌)

장바구니 항목(`substitution_cart_items`)의 한 줄은 **"대체 결정"** 이다:

```
원수종(무엇을 대체하는가) → 대체수종(무엇으로 바꾸는가)  + 현장 수량·단가 스냅샷
```

절감 계산(개선 하자율, 예비비 절감)이 **현장의 식재 수량·단가**에 의존하므로,
항목에는 `original_species_name`(원수종)과 `site_id`(현장)가 **필수**다.

그런데 **species-finder의 출력에는 이 둘이 없다.**

| | simulation 화면 | species-finder 화면 |
|---|---|---|
| 시작점 | **현장 선택** → 그 현장의 식재목록 | 조건 입력(계절·지역·목적·규격) |
| 원수종 | 식재목록의 고/중위험 수종 = **명확** | **없음** (무엇을 대체하는지 모름) |
| 출력 | 원수종별 대체 후보 | "이 조건에 좋은 수종 Top5" (추천 자체) |
| 수량·단가 | 식재행에 있음 | **없음** |

근거:
- finder 출력 구조: `species-finder-tab.tsx:135-144` (`RecommendResult`에 원수종/현장/수량 없음)
- finder는 현장 비종속: `species/page.tsx:7-30` (sites를 아예 로드하지 않음)
- 장바구니 항목 필수 필드: `actions/cart-types.ts`의 `CartItemInput`
  (`originalSpeciesName`, `quantity`, `unitPrice` 등)

**즉 finder에서 바로 담으려면, 어딘가에서 "현장 + 원수종"을 추가로 받아야 한다.**
이건 단순 버튼 추가가 아니라 finder의 정체성("탐색 도구"인가 "설계 입력 도구"인가)에 대한 결정이다.

---

## 2. 세 가지 방향

### 방향 A — 현장+원수종 선택 모달 (원래 계획안)

finder 추천 결과의 [담기] 클릭 시 모달을 띄워:
1. 현장 선택 (드롭다운)
2. 그 현장의 고/중위험 원수종 선택 (드롭다운)
3. → finder가 추천한 수종을 **대체수종**으로 하여 그 현장 카트에 담기

```
┌─ finder 추천 결과 ──────────────┐     ┌─ 담기 모달 ──────────┐
│ 1위 주목      12.3%   [담기]──┼───▶ │ 현장:  [만촌 ▼]       │
│ 2위 소나무    14.1%   [담기]   │     │ 원수종:[서양측백 ▼]   │
│ 3위 ...                        │     │ → 주목으로 대체       │
└────────────────────────────────┘     │        [취소] [담기]  │
                                        └──────────────────────┘
```

- **장점**: 계획대로. finder를 "설계 입력(forward hook)"의 입구로 격상 — 본래 의도의
  "예측 → 설계반영" 루프에 부합.
- **단점**: species 페이지에 sites + 현장별 plantings 로드를 추가해야 함
  (`species/page.tsx`, `species-tabs.tsx`, `species-finder-tab.tsx` 3개 파일 + 모달 신규).
  사실상 simulation의 데이터 흐름을 finder에도 복제. **중간 밀도 작업.**
- **구현 범위**: 신규 모달 컴포넌트, sites/plantings 쿼리, props 스레딩, 담기 액션 연결.

### 방향 B — finder는 제외, simulation에서만 담기 (현 상태 유지)

finder는 지금처럼 **순수 탐색 도구**로 두고, 담기 경험은 simulation 화면에 집중.

- **장점**: 추가 작업 0. 이미 완성된 핵심 루프로 충분. 충돌을 우회(원수종·현장 맥락이
  이미 있는 곳에서만 담음). YAGNI에 가장 부합.
- **단점**: "추천을 보다가 바로 담는" 이커머스식 흐름은 finder에 없음.
  사용자는 finder에서 좋은 수종을 본 뒤 simulation으로 가서 다시 담아야 함.
- **구현 범위**: 없음 (Task 7 생략, Task 8로 마무리).

### 방향 C — finder에 가벼운 링크만

finder 추천 결과에 [시뮬레이터에서 담기] 버튼 → 추천 수종명을 쿼리파라미터로
simulation 화면으로 이동. 담기 자체는 simulation에서 수행.

```
finder 결과 [시뮬레이터에서 담기] ──▶ /simulation?suggest=주목
                                      (simulation이 해당 수종 하이라이트)
```

- **장점**: 두 화면을 잇되 데이터 복제 없음. finder는 탐색, simulation은 결정으로
  역할 분리 유지. 경량.
- **단점**: 진짜 "한 자리에서 담기"는 아님(화면 전환 발생). simulation에서 쿼리파라미터를
  받아 처리하는 약간의 로직 필요.
- **구현 범위**: finder 결과에 링크 버튼 + simulation에서 `?suggest=` 수신 처리(소).

## 4. 추천

**핵심 가치(예측 결과를 설계 결정으로 잇기)는 이미 simulation 루프로 닫혔다.**
finder 연동은 "있으면 좋은" 확장이지 루프의 전제가 아니다.

- 지금 마무리가 목적이면 → **B** (finder 제외, 현 상태로 완성).
- 두 화면 연결을 원하되 가볍게 → **C** (링크).
- finder를 본격 "설계 입력 입구"로 키울 계획이면 → **A** (단, 별도 작업 단위로).

> 어느 쪽이든 핵심 장바구니 기능은 영향받지 않는다. 이 결정은 finder의 역할에 대한 것이다.

- [x] 사유: 핵심 가치(예측 결과를 설계 결정으로 잇는 루프)는 simulation 화면의 담기→비교→교체→확정
  루프로 이미 완결됨. finder 연동은 루프의 전제가 아니라 "있으면 좋은" 확장이며, species 페이지에
  sites/plantings 데이터 흐름을 복제해야 해 회귀·복잡도 위험이 있다. YAGNI 원칙에 부합.
- [x] 진행 시점: **나중에 별도 작업** — finder를 본격 "설계 입력 입구"로 키울 계획이 생기면
  방향 A(모달)를 별도 작업 단위로 검토한다. 그 전까지 finder는 순수 탐색 도구로 유지.

> **연관규칙 분석(Support/Confidence/Lift)** 이라는 별도 기능임이 확인됨(향후 별도 작업).
