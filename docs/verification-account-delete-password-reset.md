# 검증 시나리오 — 계정 소프트 삭제 + 비밀번호 재설정

> 대상 기능: 사용자 관리 탭의 (1) 계정 삭제(소프트), (2) 관리자 비밀번호 재설정 링크 발송, (3) 사용자 비밀번호 재설정 전용 페이지(`/reset-password`).
> 권한: 두 기능 모두 **슈퍼관리자(superadmin) 전용**.
> 브라우저 자동화(Playwright)는 마지막 "최종 자동 검증"에서 1회만 사용(메모리 정책: 중간 호출 최소화).

---

## 0. 사전 조건 (반드시 먼저 확인)

| 항목 | 확인 내용 | 미충족 시 증상 |
|---|---|---|
| **DB 마이그레이션** | `add_profiles_deleted_status.sql` 적용 완료 (status에 'deleted' 허용) | 삭제 클릭 시 "삭제 실패: ... violates check constraint" 토스트 |
| **Supabase Redirect URL** | `https://landscapingcs.vercel.app/reset-password`, `http://localhost:3000/reset-password` 등록 | 재설정 메일 링크 클릭 시 Site URL로 폴백되어 `/reset-password` 미도착 |
| **Supabase 이메일 발송** | Auth 이메일/SMTP 설정 활성 | 발송 토스트는 떠도 메일 미수신 |
| **테스트 계정** | superadmin 1개 + 삭제/재설정 대상 일반 계정 1개 | - |

> 마이그레이션 적용 여부 확인 SQL:
> ```sql
> SELECT pg_get_constraintdef(oid) FROM pg_constraint
> WHERE conrelid = 'profiles'::regclass AND contype = 'c'
>   AND pg_get_constraintdef(oid) LIKE '%status%';
> ```
> 결과에 `'deleted'`가 포함되어야 함.

## 1. 권한 가드 검증 (UI 노출)

| # | 시나리오 | 기대 결과 |
|---|---|---|
| 1-1 | **superadmin**으로 로그인 → 설정 → 사용자 탭 | 각 사용자 행(삭제됨 제외)에 **'비밀번호 재설정'**, **'삭제'** 버튼 노출 |
| 1-2 | **admin**(슈퍼관리자 아님)으로 로그인 → 사용자 탭 | 두 버튼 **미노출** (승인/비활성화/재활성화만) |
| 1-3 | 일반 사용자(user) | 사용자 탭 자체 접근 불가(설정 페이지 isAdmin=false) |

> 서버 측 가드도 함께 확인: admin 계정에서 직접 `POST /api/user-management {action:'delete'}` 호출 시 **403 "권한이 없습니다."** (UI 우회 차단). resetPassword도 동일.

## 2. 계정 소프트 삭제

| # | 시나리오 | 기대 결과 |
|---|---|---|
| 2-1 | superadmin이 대상 일반 계정 행의 **'삭제'** 클릭 | `window.confirm` "정말 삭제하시겠습니까? ..." 표시 |
| 2-2 | confirm **확인** | 토스트 "계정이 삭제되었습니다." + 해당 **행이 목록에서 즉시 사라짐**, '전체 사용자' 카운트 -1 |
| 2-3 | confirm **취소** | 아무 변화 없음 (API 미호출) |
| 2-4 | **본인(superadmin) 계정** 삭제 시도 | 토스트 "본인 계정은 삭제할 수 없습니다." (API 400, 목록 변화 없음) |
| 2-5 | 삭제된 계정으로 **로그인 시도** | 로그인 폼에 "삭제된 계정입니다. 관리자에게 문의하세요." (대시보드 진입 차단) |
| 2-6 | 새로고침 후 사용자 탭 재확인 | 삭제된 사용자가 **목록에 보이지 않음**(숨김 처리). DB에는 status='deleted'로 보존됨 |

> **설계(숨김 처리)**: `settings/page.tsx` 사용자 조회에 `.neq('status','deleted')` 적용 → 삭제된 계정은 목록·카운트에서 제외. 데이터는 DB에 `status='deleted'`로 **보존**(소프트 삭제, 복구 가능)되며 화면에서만 숨김. 클라이언트도 삭제 즉시 행을 제거해 새로고침 전후 동작이 일치.

## 4. 비밀번호 재설정 — `/reset-password` 페이지

| # | 시나리오 | 기대 결과 |
|---|---|---|
| 4-1 | 메일 링크 클릭 → `/reset-password` 도착 | 잠깐 "재설정 링크를 확인하는 중..." → recovery 세션 감지되면 **새 비밀번호 입력 폼** 표시 |
| 4-2 | 비밀번호 8자 미만 입력 후 제출 | "새 비밀번호는 8자 이상이어야 합니다." (제출 안 됨) |
| 4-3 | 두 입력 불일치 | "비밀번호가 일치하지 않습니다." |
| 4-4 | 유효한 비밀번호 일치 입력 → 변경 | "비밀번호가 변경되었습니다." → 2초 후 `/login`으로 이동 |
| 4-5 | **새 비밀번호로 로그인** | 정상 로그인 (대시보드 진입) |
| 4-6 | **링크 없이** `/reset-password` 직접 접근 (recovery 세션 없음) | "재설정 링크를 확인하는 중..." 안내 + 로그인 이동 링크만 표시(비밀번호 폼 미노출) |
| 4-7 | 만료/무효 링크로 접근 | 4-6과 동일(폼 미노출) |

## 6. 최종 자동 검증 (Playwright — 1회만)

로컬 `npm run dev`(localhost:3000) 기준, superadmin 세션으로 아래 핵심 경로만 자동 확인:

1. `/login` → superadmin 로그인 → `/settings` 사용자 탭 진입.
2. 대상 테스트 계정 행에 '삭제'·'비밀번호 재설정' 버튼 존재 확인(스냅샷).
3. '삭제' 클릭 → confirm 수락 → 상태 뱃지 '삭제됨' 변경 확인.
4. 로그아웃 → 삭제된 계정으로 로그인 시도 → "삭제된 계정입니다" 메시지 확인.
5. (선택) admin 계정 로그인 → 사용자 탭에 두 버튼 미노출 확인.

> 메일 발송/수신·실제 `/reset-password` recovery 토큰 흐름은 Supabase 메일·토큰에 의존하므로 자동화 대상에서 제외하고 **수동(3,4장)** 으로 검증.
> Playwright 호출은 위 1회 시퀀스로 끝낸다(중간 반복 호출 금지 — 메모리 정책).
